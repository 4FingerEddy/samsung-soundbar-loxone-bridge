# Security Policy

## Intended deployment

Samsung Soundbar Loxone Bridge is intended for trusted local networks only.

Do not expose the bridge directly to the public internet. If you route it through a reverse proxy, VPN or firewall, restrict access to trusted clients.

## Authentication

All control/status endpoints except `/health` require `BRIDGE_AUTH_TOKEN`.

For non-loopback bindings, the application refuses to start when `BRIDGE_AUTH_TOKEN` is empty. This is deliberate.

Supported auth modes:

- `query_or_header`: accept query-string token and bearer-header token.
- `header_only`: accept only bearer-header token.
- `query_only`: accept only query-string token.

Loxone Virtual HTTP commands often require query-token URLs. That means the token may be visible in Loxone configuration and local logs. Treat the bridge token as local and rotatable.

## Samsung AccessToken

The Samsung local API AccessToken is generated by the bridge with `createAccessToken`, cached in memory, and sent only to the soundbar local API. Users should not copy it into Loxone or deployment files.

## TLS

Many Samsung local IP-control endpoints use local/self-signed certificates. The default `SOUNDBAR_VERIFY_SSL=false` is intended for LAN-only deployments. If your environment supports certificate validation or pinning, prefer that.

## Reporting issues

Please open a GitHub issue with:

- affected version/commit
- deployment mode
- sanitized config
- sanitized logs
- reproduction steps

Never include real bridge tokens, Samsung AccessTokens, private keys or sensitive network diagrams in public issues.
